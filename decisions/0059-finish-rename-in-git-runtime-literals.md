# ADR-0059: Finish the rename in git branch/commit/label runtime literals

**Status**: Accepted
**Date**: 2026-07-20

## Context

[ADR-0055](0055-rename-everflow-to-syntropy.md) renamed the project from
everflow to syntropy across the module path, binary name, control-verb
prefix, and config directory. It missed a set of runtime string literals
in `internal/refactorsweep/workflow.go` and `internal/git/git.go`: the
git branch name prefix (`everflow/plan/...`, `everflow/<runID>/<unitID>`),
the commit message template (`"everflow %s: %s"` / `"Generated by
everflow run %s."`), the MR labels (`"everflow"`, `"everflow:<runID>"`),
and a stderr log line in the git wrapper (`"everflow git: skipped ..."`).
These are write-only outputs — new Runs kept stamping "everflow" into
every branch, commit, and MR they created, well after the rest of the
codebase had moved to "syntropy".

## Decision

Rewrite the four literals to use "syntropy":

- `branchName()` and the planning-worktree branch in `workflow.go`:
  `syntropy/<short-runID>/<unitID>` and `syntropy/plan/<short-runID>`.
- The commit message template in `work()` and in `buildCommitMessage()`:
  `"syntropy %s: %s"` / `"Generated by syntropy run %s."`.
- The MR labels passed to `CreateMR`: `"syntropy"` and
  `"syntropy:<short-runID>"`.
- The stderr line in `git.go`'s untracked-binary-file skip path:
  `"syntropy git: skipped ..."`.

No dual-recognition or backward-compat read path is needed: none of
these strings are ever pattern-matched back out of git or the provider
API by this codebase — they are produced once and only consumed by
humans reading branch/commit/label text. Old branches and MRs created
under the "everflow" name before this change are historical artifacts,
consistent with ADR-0055's treatment of pre-rename branch names.

Test literals in `workflow_test.go`, `planner_test.go`, and `git_test.go`
that mirrored the old convention (branch name fixtures, plan-branch
prefix assertions, a git test-author identity, a test file's `NEW.md`
content) were updated to match.

## Alternatives considered

- **Leave it, since old branches/labels aren't functionally broken.**
  Rejected — the whole point of ADR-0055 was that the project no longer
  goes by "everflow"; every new Run continuing to stamp that name into
  its own git history and MR labels is the rename being incomplete, not
  a cosmetic nit.
- **Support recognizing both `everflow/...` and `syntropy/...` branch
  prefixes going forward.** Rejected — there is no code path that reads
  branch names or labels back to match against a prefix, so dual
  recognition would be speculative machinery for a need that doesn't
  exist.

## Consequences

- All new Runs from this point on produce `syntropy/...` branches,
  `syntropy <runID>: ...` commit messages, `syntropy` / `syntropy:<runID>`
  MR labels, and the `syntropy git: ...` stderr line.
- Branches, commits, and MRs created by Runs before this change keep
  the `everflow` name baked in; they are not retroactively renamed,
  matching ADR-0055's precedent for historical artifacts.
